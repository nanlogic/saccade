# Saccade Launch Kit

Status: preparation only

Use these files to prepare the first public Saccade article and its Hacker News
submission. They do not authorize publication and they do not turn dogfood
results into release claims.

## Files

1. [`PUBLICATION_BACKLOG.md`](PUBLICATION_BACKLOG.md) lists the core web pages
   and the ordered article series with evidence gates.
2. [`WEB_BLOG_RELEASE_CHECKLIST.md`](WEB_BLOG_RELEASE_CHECKLIST.md) tracks the
   pages, evidence, media, metadata, and release gates needed before publishing.
3. [`MACOS_PUBLIC_BUILD_CHECKLIST.md`](MACOS_PUBLIC_BUILD_CHECKLIST.md) records
   the signed-build, notarization, staple, Gatekeeper, and clean-machine path.
4. [`MACOS_BUILD86_RELEASE_REPORT.md`](MACOS_BUILD86_RELEASE_REPORT.md) records
   the notarized Build 86 candidate, checksums, passed gates, and remaining
   clean-machine work.
5. [`ARTICLE_BROWSER_RECEIPTS.md`](ARTICLE_BROWSER_RECEIPTS.md) is the first
   English technical article draft.
6. [`HACKER_NEWS_LAUNCH.md`](HACKER_NEWS_LAUNCH.md) separates the first regular
   link submission from a later Show HN launch.

## Recommended order

1. Produce one sanitized public evidence pack from a real installed build.
2. Publish a small evidence page and the article on an owned domain.
3. Verify the live page, links, metadata, and mobile layout.
4. Submit the article as a regular Hacker News link using its original title.
5. Prepare a Show HN only after readers can install and try Saccade without a
   signup or private handoff.

The repository remains the source of truth for code. Public claims should link
to a dated evidence pack rather than a historical milestone file.
